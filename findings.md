## 조사 결과

### 기준 문서 해석

- `file-based-planning-workflow.md`는 `tasks.md`, `findings.md`, `progress.md`를 분리해서 장기 기억처럼 사용하도록 안내한다.
- `planning-with-files.md`는 복잡한 작업에서 목표를 잃지 않도록 계획 파일을 지속적으로 갱신하는 흐름을 강조한다.
- 두 문서를 함께 보면 핵심은 세 가지다.
  - 계획은 `tasks.md`에서 관리한다.
  - 중요한 발견과 결정은 `findings.md`에 축적한다.
  - 실제 작업 이력은 `progress.md`에 세션 단위로 남긴다.

### 프로젝트 주제

- 현재 프로젝트 주제는 `ChatGPT로 유튜브 동영상 쉽게 제작하기`이다.
- 이 주제는 단순한 아이디어 발상이 아니라 실제 제작 과정을 더 쉽게 만드는 흐름 설계와 밀접하다.
- 따라서 단일 문서 하나보다도 단계별 제작 흐름과 재사용 가능한 템플릿이 중요할 가능성이 높다.

### 대상 사용자와 첫 실험 주제

- 첫 대상 사용자는 `초보 유튜버`로 정했다.
- 첫 실험 주제는 `30대라면 반드시 먹어야 하는 영양제 3가지`로 정했다.
- 따라서 초반 산출물은 전문가용 제작 시스템보다 초보자도 따라할 수 있는 간단한 기준표와 템플릿 형태가 더 적합하다.

### 유튜브 제작 흐름 초안 분석

- 사용자가 제안한 기본 흐름은 다음 3단계다.
  - 주제 선정
  - 주제에 맞춘 유튜브용 대본 작성
  - 생성된 대본으로 동영상 제작
- 이 3단계는 큰 흐름으로는 자연스럽고, 최소 실행 흐름으로도 충분히 타당하다.
- 다만 실제 제작 품질을 높이려면 `대본 작성`과 `동영상 제작` 사이에 최소 1개의 중간 단계가 추가되는 편이 좋다.
- 특히 다음 요소가 빠지기 쉽다.
  - 주제 검증: 검색 수요, 타깃 관심사, 경쟁 영상 관점 확인
  - 대본 검수: 말하기 자연스러움, 길이, 첫 10초 훅, 마무리 콜투액션 점검
  - 장면 설계: 어떤 문장에 어떤 화면, 자막, 이미지, B-roll을 넣을지 정리
- 따라서 현재 3단계는 방향은 맞지만, 실제 실행용 프로세스로는 약간 거친 초안에 가깝다.

### 대화 기록 운영

- 사용자는 세션이 바뀌더라도 이전 질문 흐름을 이어갈 수 있도록 대화 내용을 파일에 남기길 원한다.
- 이를 위해 `CHATS.md`를 대화 히스토리 기록용 파일로 추가한다.
- `CHATS.md`에는 주요 질문, 요청, 이에 대한 처리 요약을 누적 기록하는 방식이 적합하다.
- 다만 `CHATS.md`는 항상 읽는 파일이 아니라, 사용자가 확인을 요청했을 때만 읽고 평소에는 기록만 갱신하는 방식으로 운영한다.

### 최종 작업 흐름 결정

- 사용자 요청에 따라 앞으로는 3단계 설명형 흐름보다 6단계 실행형 흐름을 기준으로 진행한다.
- 확정한 6단계 버전은 다음과 같다.
  - 1. 주제 선정
  - 2. 주제 검증 및 영상 방향 정리
  - 3. 유튜브용 대본 작성
  - 4. 대본 검수 및 수정
  - 5. 장면 구성표 작성
  - 6. 동영상 제작
- 이 구조는 아이디어 단계와 실제 제작 단계를 이어주는 중간 작업을 포함하므로, 결과물 품질과 재사용성 측면에서 더 안정적이다.

### 다음 실행 포인트

- 6단계 흐름이 정해졌다면 가장 먼저 해야 할 일은 1단계와 2단계를 실제로 시작할 수 있게 만드는 것이다.
- 즉, 바로 영상을 만들기보다 아래 3가지를 먼저 정해야 한다.
  - 어떤 사람을 위한 프로젝트인지
  - 어떤 주제를 먼저 다룰지
  - 첫 번째 산출물을 무엇으로 만들지
- 현재 시점에서 가장 적절한 첫 산출물은 전체 앱 구현보다 `주제 선정 기준 + 주제 검증 체크리스트 + 대본 작성 입력 양식` 같은 문서형 초안일 가능성이 높다.

### 벤치마킹 방식에 대한 판단

- 비슷한 유형의 영상에서 잘 나온 사례를 참고하는 것은 좋은 방법이다.
- 다만 `조회수가 잘 나온 영상의 대본을 가져와서 그것을 기반으로 내 대본을 만든다`는 방식은 그대로 쓰기에는 한계가 있다.
- 이유는 다음과 같다.
  - 특정 영상 표현을 그대로 따라가면 저작권과 유사성 문제가 생길 수 있다.
  - 잘 나온 이유가 대본 자체보다 채널 신뢰도, 썸네일, 제목, 업로드 시점 때문일 수도 있다.
  - 초보 유튜버에게 필요한 것은 대본 복사가 아니라 `왜 잘 먹히는지`를 구조로 이해하는 것이다.
- 따라서 더 나은 방법은 `스크립트 수집`보다 `구조 벤치마킹`이다.
- 추천 방식은 다음 순서다.
  - 1. 유사 영상 5~10개를 고른다.
  - 2. 제목, 썸네일 문구, 도입부 훅, 정보 배열 방식, 마무리 방식을 분석한다.
  - 3. 공통 구조를 뽑아 템플릿으로 만든다.
  - 4. 그 템플릿에 내 주제와 내 톤을 넣어 새 대본을 생성한다.
- 즉, `잘된 영상의 대본을 복사해서 바꾸는 방식`보다 `잘된 영상의 구조를 추출해 내 대본을 새로 생성하는 방식`이 더 좋다.

### 실제 수집 단계에서의 제약

- 유튜브 링크만으로는 현재 작업 환경에서 항상 제목, 자막, 본문 구조를 안정적으로 추출할 수 있는 것은 아니다.
- 따라서 사용자가 영상 링크와 함께 핵심 정보 일부를 같이 주는 방식이 가장 실용적이다.
- 특히 아래 정보가 있으면 구조 분석을 바로 시작할 수 있다.
  - 영상 제목
  - 도입부 10~30초 내용
  - 본문 전개 요약
  - 마무리 멘트

### 001, 002, 003 대본 구조 공통점

- 세 영상 모두 초반에 강한 문제 제기나 경고형 훅을 사용한다.
- 본문은 `순위`, `분류`, `리스트` 형식으로 전개한다.
- 각 항목마다 이유와 주의점을 짧게 붙여 신뢰감을 만든다.
- 마지막에는 시청자 행동 유도나 다음 콘텐츠 연결로 마무리한다.
- 따라서 우리 대본도 `강한 도입 -> 3가지 항목 제시 -> 항목별 이유 -> 주의사항 -> 정리 -> CTA` 구조로 작성하는 것이 적절하다.

### 첫 대본 초안 보강 판단

- 첫 초안은 구조상 문제는 없었지만 영상 길이 기준으로는 짧은 편이었다.
- 또한 영양제 이름만 소개하고 왜 필요한지 설명하는 밀도가 부족했다.
- 따라서 10분 내외 영상에 맞추려면 각 영양제마다 아래 설명이 추가되어야 한다.
  - 어떤 역할로 많이 알려져 있는지
  - 왜 30대에서 중요도가 올라가는지
  - 어떤 생활 패턴에서 필요성을 의심할 수 있는지
  - 주의할 점은 무엇인지
- 이 기준에 맞춰 대본을 설명형으로 확장하는 것이 적절하다.

### 유사 영상 분석표 템플릿 목적

- `유사 영상 분석표 템플릿`의 목적은 잘된 영상의 문장을 베끼는 것이 아니다.
- 핵심 목적은 성과가 좋은 영상의 구조적 특징을 추출하는 것이다.
- 이 템플릿을 사용하면 다음이 가능하다.
  - 제목과 썸네일의 클릭 유도 패턴 파악
  - 도입부 훅과 본문 전개 방식 분석
  - 신뢰 형성 방식과 마무리 구조 정리
  - 우리 주제에 맞는 대본 생성 입력값 추출
- 즉, 이 템플릿은 `대본 복사용`이 아니라 `대본 설계용`이다.

### 사용자가 지금 준비할 입력 자료

- 사용자가 반드시 영상 5~10개의 `전체 대본`을 모두 가져와야 하는 것은 아니다.
- 가장 좋은 입력 자료는 아래 순서다.
  - 1. 영상 제목
  - 2. 영상 링크
  - 3. 썸네일 문구
  - 4. 도입부 10~30초 정도의 내용
  - 5. 본문이 어떻게 전개되는지에 대한 요약
  - 6. 마무리 멘트나 행동 유도 방식
- 만약 전체 자막이나 대본이 있으면 더 좋지만, 없어도 구조 분석은 충분히 가능하다.
- 즉, 사용자가 지금 해야 할 일은 `좋은 영상 5~10개를 고르고, 구조 분석에 필요한 핵심 정보만 모아오는 것`이다.

### 유튜브 대본 추출 도구 판단

- 자막이 이미 있는 영상이라면 가장 쉬운 방법은 유튜브 자체 `스크립트 보기` 기능을 쓰는 것이다.
- 자막을 파일로 저장하거나 여러 영상을 반복 처리하려면 `yt-dlp` 같은 도구가 더 적합하다.
- 자막이 없거나 자동 자막 품질이 낮다면 다운로드한 음성을 `Whisper`류 음성 인식 도구로 다시 전사하는 방법이 유용하다.
- 따라서 추천 우선순위는 다음과 같다.
  - 1. 유튜브 기본 스크립트 보기
  - 2. `yt-dlp`로 자막 다운로드
  - 3. `Whisper`로 직접 전사
- 이 프로젝트 기준에서는 `유튜브 기본 스크립트 보기` 또는 `yt-dlp`가 1차 선택이고, 필요할 때만 `Whisper`를 보조적으로 쓰는 것이 효율적이다.

### 현재 작업 상태

- 현재 저장소는 구현보다 계획 정리 단계에 더 가깝다.
- 이미 `AGENTS.md`, `tasks.md`, `findings.md`, `progress.md`가 루트에 존재한다.
- 세션 간 대화 맥락 유지를 위해 `CHATS.md`를 함께 관리하기로 했다.
- 앞으로 작업을 진행할 때는 결과를 세 파일과 대화 기록 파일에 주기적으로 반영해야 한다.

| 유사 영상 분석표 템플릿을 먼저 만든다 | 구조 벤치마킹 결과를 일관된 형식으로 모으기 위함이다 |

## 발생한 이슈

### 1. 초기 프로젝트 정보 부족

**문제**: 처음에는 프로젝트 파일과 실제 산출물 방향이 충분히 드러나지 않았다.

**영향**: 구현 중심 계획을 세우기보다 먼저 계획 구조와 운영 규칙을 정비해야 했다.

**해결**: 파일 기반 계획 구조를 먼저 만들고, 이후 프로젝트 주제와 로컬 기준 문서를 반영해 수정했다.

### 2. 기준 문서가 확장됨

**해결**: 두 문서의 공통 원칙을 묶어 `tasks.md`, `findings.md`, `progress.md`를 재정렬했다.

### 3. 인코딩 문제로 파일 수정 실패

**문제**: 일부 마크다운 파일의 인코딩이 깨져 자동 수정 도구가 내용을 읽지 못했다.

**해결**: 파일 전체를 UTF-8 한글 기준으로 다시 저장하며 최신 내용을 반영했다.
